# AIOps 智能运维平台 - 根因分析架构设计

> 本文档基于多次架构讨论整理，记录了 AIOps 平台的核心设计理念、根因分析技术难点与未来演进方向。

---

## 一、架构共识与方向

### 1.1 核心判断

**结论：偏差不大，属于"同一路线、重心不同"。**

| 视角 | 描述 |
|------|------|
| 当前方案 | 日志驱动 + 两轮处理 + Agent 执行 |
| 理想方向 | 统一观测信号 + 事故对象化 + AI 辅助决策 |

**一句话总结**：
> 你的方案已经是"AIOps 平台雏形"；需要从"日志二轮理解平台"推进成"以 Incident 为中心的统一证据与决策平台"。

### 1.2 高度一致的三个观点

1. **没有走"所有日志都丢给大模型"这条死路**
   - 先低成本规则处理
   - 再把少量高价值数据交给 AI
   - 方向正确

2. **应急处理和复盘分析拆成两条链路**
   - 实时链路：稳、快
   - 复盘链路：深语义分析

3. **闭环运维**
   - 包含执行层、通知、审批流
   - 不是"有个会说话的机器人"那么简单

## 二、两轮数据处理设计

### 2.1 为什么需要两轮？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          两轮数据处理的必要性                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  原始日志                                                                    │
│    │                                                                       │
│    ▼                                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │ 第一轮：规则清洗 (Logstash)                                           │  │
│  │  • 正则匹配：提取时间、IP、错误码                                    │  │
│  │  • 去重过滤：重复超过1000次的日志丢弃                               │  │
│  │  • 格式标准化：转JSON/XML                                             │  │
│  │  • 脱敏处理：手机号、身份证等敏感信息掩码                          │  │
│  │                                                                     │  │
│  │ 输出：结构化但语义模糊的"死数据"                                     │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│    │                                                                       │
│    ▼                                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐  │
│  │ 第二轮：语义增强 (LLM)                                               │  │
│  │  • 意图识别：这个日志代表什么事件？（连接超时/内存泄漏/磁盘满）      │  │
│  │  • 关键实体提取：提取关键信息（表名、接口路径、用户ID）             │  │
│  │  • 关联分析：这条日志和之前哪些日志相关？                            │  │
│  │  • 语义标注：标记严重程度、影响范围                                  │  │
│  │                                                                     │  │
│  │ 输出：语义丰富的"活数据"                                              │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 "清洗"vs"增强"的命名之争

**核心观点**：建议把"二轮清洗"改成"二轮语义增强"

| 对比项 | 清洗（Cleaning） | 语义增强（Enrichment） |
|--------|-----------------|----------------------|
| 含义 | 标准化、纠错、去噪、脱敏、字段对齐 | 意图分类、实体提取、摘要、假设生成 |
| 执行者 | 确定性逻辑（正则、规则） | LLM（理解+补充） |
| 输出性质 | "定义事实" | "补充语义" |
| 稳定性 | 高 | 需要约束 |

**结论**：
> LLM 应该负责"理解"和"补充语义"，不应该负责"定义事实"。

### 2.3 一轮输出的字段标准

第一轮输出必须对齐到**统一字段标准**，推荐：

- **Elastic Common Schema (ECS)**：Elasticsearch 中事件数据的通用字段
- **OpenTelemetry Semantic Conventions**：traces、metrics、logs 等信号的统一命名
- **JSON Schema**：约束 JSON 的结构、类型和校验规则

```json
{
  "@timestamp": "2026-03-10T12:34:56Z",
  "service.name": "payment-service",
  "host.name": "node-12",
  "trace.id": "abc123",
  "span.id": "def456",
  "event.kind": "error",
  "event.category": "database",
  "log.level": "error",
  "message": "Connection pool exhausted"
}
```

## 三、统一观测中心设计

### 3.1 从"日志中心"到"统一观测中心"

**关键差异**：

| 视角 | 日志中心 | 统一观测中心 |
|------|---------|-------------|
| 数据优先级 | 日志为主 | traces / metrics / logs / change 同级 |
| 根因定位 | 日志解释系统 | 多信号关联分析 |
| 上下文 | 不足 | 完整 |

**必须提升到一级公民的数据**：

- `trace_id / span_id`：描述请求穿过系统的完整路径
- 服务拓扑 / 依赖图：服务间依赖关系
- 最近变更事件：发布、配置、扩缩容、开关
- dependency metrics：下游依赖的延迟、错误率、吞吐

## 四、Incident 中心设计

### 4.1 为什么需要事故收敛？

**问题**：一个事故可能产生 1000+ 告警

**传统场景**：
```
服务A异常 → 接口超时 → Redis慢 → 数据库连接高 → CPU高 → Pod重启
监控系统会报：100个报警
运维要自己推理
```

**AI 做法**：告警聚合 + 拓扑分析
```
100个报警 → AI分析依赖关系 → 聚合为1个Incident
Root: Redis latency
```

## 五、根因分析为什么这么难？

### 5.1 五大技术难点

| 难点 | 描述 |
|------|------|
| **多数据源关联** | metrics / logs / traces / topology / change events 格式不同、粒度不同 |
| **时间同步** | 不同数据源时间戳不同步，需找到共同时间窗口 |
| **因果推断** | 多个事件同时发生，需区分相关性与因果性 |
| **数据噪音** | 日志丢失、指标异常、监控配置错误 |
| **上下文理解** | 需要理解业务上下文、部署历史、用户行为 |

### 5.2 工程挑战

1. **准确性**
   - 训练数据覆盖不全时，推理结果可能不准确
   - 需要大量标注数据

2. **实时性**
   - 运维需要快速响应
   - 传统 ML 模型训练和优化时间长

3. **可解释性**
   - AI 结论必须可审计
   - 深度学习是"黑箱"，难以解释

4. **跨层级根因**
   - 根因可能不在单一层级
   - 应用问题 → 系统资源 → 硬件

## 六、关键技术决策

### 6.1 二轮小模型选型

**建议：按任务拆模型，不是选一个**

| 任务类型 | 模型选择 | 说明 |
|---------|---------|------|
| 模板分类、实体抽取、标签补齐、严重度粗分 | 本地小模型 (qwen2.5:0.5b) | 高频、窄任务、结构化输出 |
| 事故摘要、跨信号关联解释、复盘报告 | API 大模型 | 深层分析 |

### 6.2 输出 Schema 定义

**必须有严格 Schema，不是自由文本**

```json
{
  "schema_version": "1.0",
  "event_id": "uuid",
  "timestamp": "2026-03-10T12:34:56Z",
  "service.name": "payment-service",
  "host.name": "node-12",
  "trace_id": "abc",
  "span_id": "def",
  "event.kind": "error",
  "event.category": "database",
  "intent": "connection_pool_exhausted",
  "severity": "high",
  "confidence": 0.82,
  "entities": {
    "db.instance": "order-db-primary",
    "http.route": "/pay/submit"
  },
  "related_change_id": "deploy-20260310-1029",
  "evidence_refs": [
    "log:es://...",
    "trace:apm://...",
    "metric:prom://..."
  ],
  "recommended_action": [
    "rollback_deploy",
    "restore_pool_size"
  ]
}
```

### 6.3 成本控制策略

**三级预算**：

| 维度 | 说明 |
|------|------|
| **按事故预算** | 一次 incident 允许消耗多少 tokens / GPU 秒 |
| **按服务等级预算** | 核心链路比普通链路更高额度 |
| **按时段预算** | 非事故时几乎不跑，事故窗口临时放宽 |

**经验值**：
- 平时：只让极少数样本进入二轮
- 告警窗口：只处理和 incident 强相关的代表样本
- 复盘阶段：可以放宽，但仍然先聚类再喂模型

### 6.4 执行分级策略

| 级别 | 风险 | 示例 | 审批要求 |
|------|------|------|---------|
| **L1** | 低、可逆、幂等 | 重启单 Pod、扩容 1 个副本、清理临时文件 | 自动执行 |
| **L2** | 中、有限影响面 | 切流、回滚某个服务、关闭开关 | 人工一键确认 |
| **L3** | 高、跨系统、不可逆 | 改数据库参数、批量重启、删缓存全集群 | 严格审批 |

**决策矩阵**：
```
        低置信度    高置信度
低风险     确认       自动
高风险     审批       确认
```

## 七、向量化存储的正确用法

### 7.1 存什么

| 推荐存入 | 不推荐存入 |
|---------|-----------|
| 事故摘要 | 全部二轮日志 |
| 根因结论 | 原始观测数据 |
| 修复动作 | 低价值重复日志 |
| Runbook 片段 | 全部告警事件 |
| 常见错误模式 | 完整日志副本 |
| 已验证的排障路径 | |

### 7.2 核心原则

> **向量库更适合存"可复用经验"，不是存"全部观察事实"。**

## 八、Agent 设计优化

### 8.1 MVP 阶段建议

**不推荐**：一开始就拆成 4 个 Agent（Planner / Analyzer / Executor / Report）

**推荐**：一个 Orchestrator + 明确的 Typed State + 若干工具节点

**何时拆分**：
- 单个上下文太大
- 排障/执行职责冲突
- 不同角色需要不同权限边界
- 复盘报告与实时处置完全不同步

## 十一、总结

### 核心结论
